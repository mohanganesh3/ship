## Offline, Low-Resource Maritime Knowledge Assistant

> **Goal**: Build a fully offline chatbot that runs on low-spec computers aboard ships *and on constrained edge devices*, trained on maritime textbooks/manuals **with all knowledge baked into the model weights**.

## 1. Problem Analysis

### Constraints
| Constraint | Detail |
|---|---|
| **Internet** | None or extremely expensive satellite connection |
| **Hardware** | Low-spec computers, 4-8 GB RAM, possibly no GPU, small SSDs |
| **Users** | Ship crew — engineers, navigators, officers |
| **Knowledge domain** | Maritime engineering, navigation, safety, regulations, equipment manuals |
| **Reliability** | Must work 100% offline, no cloud dependencies |
| **Maintenance** | Cannot be easily updated at sea (weeks/months between ports) |
| **Interface** | Simple chat UI, usable by non-technical crew |

### What We're Building
A **self-contained AI system** that:
- Runs entirely on a ship's local computer (4-8 GB RAM, CPU-only)
- Answers questions from maritime textbooks and manuals
- Provides accurate, grounded answers with source citations
- Works without any internet connection
- Has a simple web-based chat interface accessible from any device on the ship's LAN

### Single-Pronged Approach: Knowledge-in-Weights

We **do not** depend on retrieval at inference time.

Instead, we:
- **Inject maritime knowledge into weights** via Continued Pretraining (CPT) on domain text.
- **Crystallize** that knowledge into *answerable* patterns via synthetic instruction tuning (SFT/distillation) generated by a strong **local** teacher model.
- **Align** behavior and refusal style with **ORPO** using preference pairs.
- Use **dynamic test-time compute routing** (“thinking” only when needed) to preserve battery and latency.

## 4. Phase 2: Knowledge Corpus Prep (for training)

### 4.1 Document Processing Pipeline (Training-Only)

We still need a robust extraction/cleaning pipeline — not to build a vector DB, but to:
1) feed **CPT** with clean domain text, and
2) feed the **teacher generator** with grounded excerpts to produce synthetic Q/A + preference data.

```
Textbooks (PDF/EPUB) → Docling/PyMuPDF → Clean text (+tables where possible)
                                                            ↓
                                         Semantic chunking (512–2048 tokens)
                                                            ↓
                    ┌───────────────┬───────────────────────┐
                    ▼               ▼                       ▼
         CPT corpus      Teacher prompts        Eval / audits
     (raw text)     (grounded excerpts)   (spot-check vs source)
```

#### Document Processing Tools (Choose One)

| Tool | Best For | License |
|------|----------|---------|
| **Docling** (Recommended) | PDFs with tables, figures, equations | MIT |
| **PyMuPDF** | Simple PDFs, fast extraction | AGPL/Commercial |
| **Unstructured** | Mixed document types | Apache 2.0 |

#### Chunking Strategy

Use **Recursive Character Chunking** with semantic awareness:
```python
# Recommended settings
chunk_size = 512       # tokens (not characters)
chunk_overlap = 64     # ~12.5% overlap
separators = ["\n\n", "\n", ". ", " "]  # paragraph → sentence → word
```

**Why 512–2048 tokens?**
- For **teacher generation**, slightly larger chunks (e.g., 1024–2048 tokens) reduce fragmentation and improve grounded Q/A.
- For **CPT**, use contiguous 2048-token blocks (or whatever the model’s max sequence length is during training).

**Advanced (2025)**: Use **Contextual Chunking** — prepend a short context description to each chunk before:
- feeding it to the **teacher** (improves grounded Q/A), and/or
- storing it as metadata for audits/evaluation.
```
"This chunk is from Chapter 5: Marine Diesel Engines, discussing 
the fuel injection timing procedure for 4-stroke engines."
[actual chunk content follows]
```
In our no-RAG setup, this is mainly about **better grounding for teacher generation** and easier audits (i.e., fewer “orphan chunks” with unclear provenance).

### 4.2 Coverage-first data design (how we avoid “RAG-level gaps”)

Because we can’t look things up at runtime, we need data design that forces memorization:

1. **Multi-angle questions per fact**
    - Generate 5–10 paraphrases and scenario variants for each key rule/number/threshold.
2. **Hard negatives / confusion pairs**
    - E.g., “global sulfur cap” vs “ECA cap”; “SOLAS II-1” vs “II-2”.
3. **Preference pairs for safety + uncertainty**
    - Chosen: careful, structured, admits uncertainty.
    - Rejected: overconfident, unsafe, or hallucinatory.
4. **DEITA-style filtering**
    - Keep the highest-quality, most diverse samples instead of maximizing raw count.

## 5. Phase 3: Model Training Pipeline

### 5.1 Training Strategy Overview

We use a **multi-stage pipeline**, progressing from general to specific:

```
Stage 1: Continued Pretraining (domain vocabulary)
    ↓
Stage 2: Instruction Tuning (QA behavior) 
    ↓
Stage 3: Reasoning Distillation (deep understanding)
    ↓  
Stage 4: Alignment (safety, helpfulness)
    ↓
Stage 5: Quantization (deployment-ready)
```

### 5.2 Stage 1: Continued Pretraining on Maritime Text

**What**: Feed raw maritime textbook text to the model as a language modeling task. This teaches the model maritime vocabulary, concepts, and relationships.

**Technique**: Standard causal language modeling (next-token prediction) on your textbook corpus.

**Method**: QLoRA (4-bit quantized LoRA) — trains with only 4-6 GB VRAM

```python
# Using Unsloth (2x faster, 70% less VRAM)
from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="Qwen/Qwen3-1.7B",
    max_seq_length=2048,
    load_in_4bit=True,  # QLoRA
)

model = FastLanguageModel.get_peft_model(
    model,
    r=16,            # LoRA rank
    lora_alpha=32,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                     "gate_proj", "up_proj", "down_proj"],
    lora_dropout=0,
    use_rslora=True,  # Rank-stabilized LoRA (2024) — better for rank > 16
)

# Train on raw textbook text
trainer = UnslothTrainer(
    model=model,
    train_dataset=maritime_text_dataset,
    args=TrainingArguments(
        per_device_train_batch_size=2,
        gradient_accumulation_steps=8,
        num_train_epochs=2,
        learning_rate=5e-5,
        bf16=True,
        output_dir="./maritime-cpt",
    ),
)
trainer.train()
```

**Data preparation**: Concatenate all textbook chapters, split into 2048-token blocks.

**Hardware**: 1x GPU with 8-16 GB VRAM (e.g., RTX 3060/4060, or cloud T4/A10G). With Unsloth + QLoRA, even a free Colab T4 works.

### 5.3 Stage 2: Supervised Fine-Tuning (Instruction Tuning)

**What**: Train the model to answer questions in chat format using maritime Q&A pairs.

**Data Generation — The Critical Step**:
You need Q&A pairs from your textbooks. Here's how to generate them:

#### Method A: Use a Large *Local* Teacher to Generate Training Data (Synthetic Data Generation)

```python
# Use a strong open-weight teacher (served locally) to generate Q&A pairs from textbook chunks
prompt = """Given this textbook excerpt, generate 5 diverse question-answer pairs 
that a ship crew member might ask. Include:
- Factual questions ("What is the minimum pressure for...")
- Procedural questions ("How do you perform...")  
- Troubleshooting questions ("What should you do if...")
- Safety questions ("What are the safety precautions for...")
- Regulatory questions ("According to SOLAS...")

Textbook excerpt:
{chunk}

Format each as:
Q: [question]
A: [detailed answer citing the textbook]
"""
```

This is **synthetic data distillation**: a large teacher reads your excerpts and emits high-signal training examples tailored for a small student.

**Important change:** We do this **fully locally** on the available 4-GPU machine (multi-day runs are fine). No paid frontier APIs.

#### Method B: Reasoning Distillation (Local “R1-style” teacher)

Instead of only generating Q&A pairs, use a strong *reasoning-capable* open-weight teacher to generate **step-by-step reasoning traces** for troubleshooting / multi-step tasks:

```python
prompt = """You are a maritime engineering expert. A crew member asks:
"{question}"

Think through this step-by-step, showing your complete reasoning process.
Then provide the final answer.

Reference material (must be obeyed): {textbook_chunk}

Format:
<think>
[Your step-by-step reasoning here]
</think>

<answer>
[Concise final answer]
</answer>
"""
```

**Why this matters**: DeepSeek showed that distilling reasoning chains from R1-70B into Qwen-1.5B-Instruct achieved **83.9% on MATH-500** — a massive jump. The small model learns *how to think*, not just *what to answer*.

**For this domain**: generate reasoning traces for troubleshooting scenarios:
```
Q: "The main engine exhaust temperature is 50°C above normal on cylinder 3"
<think>
High exhaust temp on one cylinder indicates incomplete combustion or 
excessive fuel injection. Let me consider the causes:
1. Fuel injector malfunction — could be stuck open or poor atomization
2. Exhaust valve not seating properly — allowing blow-by
3. Turbocharger issue — but this would affect all cylinders
4. Scavenge port blockage — restricting air supply to that cylinder
Since only cylinder 3 is affected, it's likely a cylinder-specific issue.
The most common cause is a faulty fuel injector...
</think>
```

#### Method C: Evol-Instruct (WizardLM Style)

Progressively make your Q&A pairs harder:
```
Simple → "What is a fuel injector?"
Medium → "Compare fuel injection timing between 2-stroke and 4-stroke marine diesel engines"
Hard → "A vessel's main engine cylinder 3 shows 50°C above normal exhaust temp with slight 
        knocking sound. The scavenge pressure is normal. Diagnose the issue and outline 
        the repair procedure following maker's guidelines."
```

#### Recommended Data Mix

| Data Type | Percentage | Source |
|-----------|-----------|--------|
| Textbook Q&A (synthetic) | 40% | Generated from your textbooks by a local teacher |
| Reasoning traces | 20% | Generated by a local reasoning-capable teacher |
| Troubleshooting scenarios | 15% | Maritime-specific problem-solving |
| Safety procedures | 10% | SOLAS, ISM Code, company procedures |
| General instruction following | 10% | Open-source datasets (FLAN, OpenAssistant) |
| Negative examples | 5% | "I don't have info about..." for out-of-scope queries |

**Target**: ~20,000–100,000 *high-quality* examples (after filtering) is a solid range for a 1.7B model when you must rely on weights-only knowledge.

#### SFT Training Code

```python
from unsloth import FastLanguageModel
from trl import SFTTrainer

# Apply chat template
def format_chat(example):
    return tokenizer.apply_chat_template([
        {"role": "system", "content": "You are a maritime knowledge assistant aboard a ship. Answer questions accurately using your training knowledge. If unsure, say so."},
        {"role": "user", "content": example["question"]},
        {"role": "assistant", "content": example["answer"]}
    ], tokenize=False)

trainer = SFTTrainer(
    model=model,
    train_dataset=maritime_qa_dataset,
    args=TrainingArguments(
        per_device_train_batch_size=2,
        gradient_accumulation_steps=4,
        num_train_epochs=3,
        learning_rate=2e-5,
        warmup_ratio=0.1,
        bf16=True,
        output_dir="./maritime-sft",
        neftune_noise_alpha=5,  # NEFTune: +2% quality for free
    ),
)
trainer.train()
```

### 5.4 Stage 3: Advanced Training — Pick What Applies

#### Option A: GRPO (Group Relative Policy Optimization) — Research context (not recommended for our <3B student)

**What**: Train the model using reinforcement learning WITHOUT needing a separate reward model. The model generates multiple answers, and the best ones are reinforced.

**Why it's revolutionary**: DeepSeek's GRPO uses 50% less VRAM than standard RLHF because there's no critic model. It's the technique behind DeepSeek-R1.

**When to use**: Generally **not** for this project’s 1.7B student. At this scale and for maritime factual QA, the safer, more reproducible approach is:

- **distill** reasoning traces via SFT on a small slice (optional), and
- use **ORPO** for preference alignment.

If you experiment with RL, restrict it to **verifiable sub-tasks** (RLVR style) where correctness can be automatically checked.

```python
from trl import GRPOTrainer, GRPOConfig

def maritime_reward(completions, prompts):
    """Reward function: checks if the answer is grounded in knowledge"""
    rewards = []
    for completion in completions:
        score = 0.0
        # Reward for citing sources
        if any(ref in completion for ref in ["Chapter", "Section", "SOLAS", "IMO"]):
            score += 1.0
        # Reward for structured answers
        if "Step" in completion or "1." in completion:
            score += 0.5
        # Penalize "I don't know" when answer exists in training data
        if "I don't know" in completion:
            score -= 0.5
        # Could also use a verifier model here
        rewards.append(score)
    return rewards

config = GRPOConfig(
    num_generations=4,  # Generate 4 answers, reinforce the best
    per_device_train_batch_size=1,
    gradient_accumulation_steps=4,
    learning_rate=1e-6,
    num_train_epochs=1,
)

trainer = GRPOTrainer(
    model=model,
    config=config,
    train_dataset=maritime_prompts,
    reward_funcs=[maritime_reward],
)
trainer.train()
```

#### Option B: ORPO (Odds Ratio Preference Optimization) ✅ Recommended

**What**: Preference optimization **without a reference model** (stable, lightweight). This is our chosen alignment layer.

**When to use**: After SFT, when you have pairs of (chosen, rejected) answers for the same prompt.

```python
from trl import ORPOTrainer, ORPOConfig

# Data format: {"prompt": "...", "chosen": "good answer", "rejected": "bad answer"}
# Recommended rejected answers: (a) early student checkpoint outputs, or
# (b) teacher-generated close-but-wrong failure modes (wrong units, missing step, ungrounded confidence).

config = ORPOConfig(
    beta=0.1,
    learning_rate=5e-6,
    per_device_train_batch_size=2,
    num_train_epochs=1,
)
trainer = ORPOTrainer(model=model, train_dataset=preference_data, ...)
trainer.train()
```

#### Option C: RLVR (Reinforcement Learning from Verifiable Rewards)

**What**: Use automatically verifiable rewards (not human preferences). Perfect for technical domains.

**When to use**: When you can programmatically check if an answer is correct (e.g., numerical calculations, procedure checklists).

```python
def verifiable_reward(question, answer, ground_truth):
    """Check if the answer contains the correct technical values"""
    # For numerical questions
    if ground_truth["type"] == "numerical":
        return 1.0 if ground_truth["value"] in answer else 0.0
    # For procedural questions  
    if ground_truth["type"] == "procedure":
        steps_found = sum(1 for step in ground_truth["steps"] if step in answer)
        return steps_found / len(ground_truth["steps"])
```

#### Option D: Self-Play Fine-Tuning (SPIN)

**What**: The model plays against itself — generating answers and then trying to distinguish its answers from ground truth answers. Iteratively improves without external data.

**When to use**: After SFT, as a no-cost improvement technique.

### 5.5 Stage 4: Alignment & Safety

For a ship chatbot, alignment means:
- **Never give dangerous safety advice** (fire, emergency procedures must be accurate)
- **Admit uncertainty** rather than hallucinate
- **Stay in domain** — don't answer unrelated questions
- **Be concise** — crew need quick answers

**Use ORPO** (Odds Ratio Preference Optimization) — combines SFT and preference alignment efficiently **without a reference model**:

```python
from trl import ORPOTrainer, ORPOConfig

config = ORPOConfig(
    beta=0.1,
    learning_rate=5e-6,
    per_device_train_batch_size=2,
    num_train_epochs=2,
)
# Needs paired data: (prompt, chosen_response, rejected_response)

**Data selection note (DEITA):** before SFT/ORPO, apply DEITA-style selection:
- Score samples for **quality** and **complexity**; keep high $q\times c$.
- Enforce **diversity** using an embedding-based representation filter (paper uses a threshold like $\tau\approx0.9$).
```

## 6. Phase 4: Quantization & Optimization

### 6.1 Quantization Strategy

After training, we need to compress the model to fit in ship hardware.

#### Recommended: GGUF Q4_K_M

| Quant Type | Size (1.7B) | RAM Needed | Quality | Speed |
|-----------|-------------|------------|---------|-------|
| Q8_0 | ~1.8 GB | ~4 GB | 99.5% of FP16 | Fast |
| **Q4_K_M** ★ | **~1.2 GB** | **~3 GB** | **98% of FP16** | **Fast** |
| Q4_K_S | ~1.1 GB | ~2.8 GB | 97% of FP16 | Fast |
| Q3_K_M | ~0.9 GB | ~2.5 GB | 95% of FP16 | Fast |
| Q2_K | ~0.7 GB | ~2 GB | 90% of FP16 | Fast |
| IQ2_M | ~0.6 GB | ~1.8 GB | 88% of FP16 | Moderate |

**Q4_K_M** is the sweet spot — virtually no quality loss, runs great on CPU.

#### Quantization Process

```bash
# Step 1: Export model to GGUF
python convert_hf_to_gguf.py ./maritime-model --outtype f16 --outfile maritime-f16.gguf

# Step 2: Quantize
./llama-quantize maritime-f16.gguf maritime-q4km.gguf Q4_K_M

# Or using Unsloth (even easier):
model.save_pretrained_gguf("maritime-model", tokenizer, quantization_method="q4_k_m")
```

### 6.2 Inference Optimization for CPU

1. **Thread Count**: Set to physical cores (not hyperthreaded)
   ```bash
   llama-cli -m maritime-q4km.gguf -t 4  # 4 physical cores
   ```

2. **Context Size**: Use small context (2048) to save RAM
   ```bash
   llama-cli -m maritime-q4km.gguf -c 2048
   ```

3. **Batch Size**: Smaller = less RAM, slightly slower
   ```bash
   llama-cli -m maritime-q4km.gguf -b 256
   ```

4. **Mmap**: Memory-map the model file — only loads needed parts
   ```bash
   llama-cli -m maritime-q4km.gguf --mmap
   ```

## 8. Phase 6: Testing & Validation

## 11. Advanced Techniques Deep Dive

### 11.1 Technique Comparison Matrix

| Technique | What It Does | VRAM Needed | Best For | Complexity |
|-----------|-------------|-------------|----------|------------|
| **Full Fine-tuning** | Updates all weights | 16+ GB | Maximum quality | Medium |
| **LoRA** | Low-rank adapters | 8-12 GB | Good quality, efficient | Low |
| **QLoRA** | LoRA + 4-bit quantization | 4-6 GB | Training on consumer GPUs | Low |
| **DoRA** | Weight-decomposed LoRA | 6-8 GB | Better than LoRA, slight overhead | Low |
| **rsLoRA** | Rank-stabilized LoRA | 4-6 GB | Higher ranks without instability | Low |
| **GaLore** | Gradient low-rank projection | 4-8 GB | Memory-efficient full fine-tuning | Medium |
| **Knowledge Distillation** | Teacher → Student transfer | 8-16 GB | Creating small from large | Medium |
| **Reasoning Distillation** | Transfer CoT reasoning | 8-16 GB | Teaching models to think | Medium |
| **ORPO** | Preference alignment (no reference model) | 6-10 GB | Better outputs without RM | Low |
| **DPO** (alt) | Preference alignment (reference-model based) | 6-10 GB | Alternative to ORPO | Low |
| **GRPO** | Group RL optimization | 6-10 GB | Research; best when rewards are verifiable | Medium |
| **RLVR** | RL from verifiable rewards | 6-10 GB | Technical/factual domains | Medium |
| **SPIN** | Self-play improvement | 6-10 GB | Iterative self-improvement | Medium |
| **Continued Pretraining** | Domain vocabulary | 4-8 GB | Teaching domain language | Low |
| **NEFTune** | Noise on token embeddings (internal) | 0 extra | Free +2% quality boost | Trivial |
| **Pruning** | Remove unnecessary weights | CPU | Making models smaller | Medium |
| **Model Merging** | Combine model strengths | CPU | Best of multiple models | Low |
| **Test-time Compute** | Think longer at inference | 0 | Better answers, slower | Low |
| **Budget Forcing** | Control reasoning length | 0 | Optimized thinking time | Low |
| **Speculative Decoding** | Small draft → large verify | 0 extra | Faster inference | Medium |
| **Contextual Chunking (training-only)** | Add context headers to chunks | 0 | Better teacher grounding + audits | Low |

### 11.2 Cutting-Edge 2025-2026 Techniques Worth Knowing

#### 1. **BitNet b1.58 — 1-bit Models (Microsoft, 2024-2025)**
- Models with ternary weights {-1, 0, 1} — no floating point math
- 0.4 GB for a 3B model, runs on bare metal CPU
- `bitnet.cpp` achieves near-lossless quality
- **For your case**: If Qwen3-1.7B is too large, a BitNet model could run on a Raspberry Pi

#### 2. **Mixture-of-Thoughts Dataset (Open R1, 2025)**
- HuggingFace's approach to training reasoning: mix multiple thinking styles
- Each Q&A pair has reasoning from different models (DeepSeek-R1, QwQ, etc.)
- The small model learns diverse reasoning strategies

#### 3. **SmolLM3's Training Recipe (HuggingFace, 2025)**
- Train on 11T tokens of curated web data
- Apply iterative preference alignment (use **ORPO** in our pipeline)
- Use model merging to combine specialist capabilities
- Result: 3B model matching GPT-3.5 on many tasks

#### 4. **Selective Layer Fine-Tuning (2025)**
- Only fine-tune the last few transformer layers
- 90% of task-specific knowledge is in the final layers
- Drastically reduces training time and prevents forgetting

#### 5. **Context Extension via YaRN/NTK (2024-2025)**
- Extend context window of small models from 2K to 32K+ tokens
- Allows including more textbook context in prompts
- Applied AFTER fine-tuning, requires no additional training

#### 6. **Continual Learning Without Forgetting (2025)**
- LoRA replay: keep a small buffer of general data while training on domain data
- Elastic Weight Consolidation (EWC): protect important weights
- Prevents catastrophic forgetting of general language abilities

### 11.3 What NOT to Worry About

| Technique | Why Skip It |
|-----------|-------------|
| RLHF (full) | Overkill for domain-specific chat; **ORPO** is simpler |
| Full pre-training from scratch | Way too expensive, base models are great starting points |
| MoE architecture changes | Too complex, existing dense models work fine at 1-3B |
| Multi-modal training | Unless you need image understanding of equipment photos |
| Online learning at sea | Too risky, model should be frozen after deployment |
